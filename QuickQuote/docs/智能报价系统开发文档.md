# QuickQuote 当前版开发文档

更新日期：2026-06-23

本文档面向继续开发者，描述当前代码真实结构。早期“LLM 生成 SQL 后估价”的方案已经不是当前主流程。

## 1. 架构概览

后端：

- FastAPI 提供接口。
- `ValuationWorkflow` 负责报价流程编排。
- MySQL 查询采购历史和订单商品库。
- 聚水潭 OpenAPI 查询商品、库存、采购入库成本。
- SQLite `brand_item_price/price.db` 查询供应商报价。

前端：

- React + TypeScript + Vite。
- SSE 消费后端报价流。
- Zustand 管理报价状态。
- Tailwind + lucide-react 组成工作台 UI。
- xlsx 在浏览器端生成导出文件。

## 2. 后端模块

```text
app/api/v1/endpoints/chat.py
app/workflow/graph.py
app/services/extract_service.py
app/services/order_goods_service.py
app/services/purchase_record_service.py
app/services/jushuitan_service.py
app/services/supplier_quote_service.py
app/services/chat_job_service.py
app/services/quote_archive_service.py
```

`chat.py` 负责把 multipart 请求转换成工作流输入，并以 SSE 方式返回阶段事件和最终结果。

`graph.py` 是主业务逻辑。新增匹配、排序、字段合并时优先从这里入手。

## 3. 工作流节点

```text
extract_stage
multi_source_match_stage
purchase_route_stage
result_stage
```

`extract_stage`：

- 合并文字、Excel 行、文件、图片。
- 调用 LLM 解析商品。
- 失败时规则兜底。

`multi_source_match_stage`：

- 并行查询订单商品库、采购表、供应商报价表。
- 并行调用聚水潭。
- 各数据源失败时记录错误，不中断整批。

`purchase_route_stage`：

- 合并候选。
- 补充采购历史。
- 补充聚水潭库存、成本、动态成本。
- 保留供应商报价字段。

`result_stage`：

- 输出 `items[].candidates[]`。
- 写入上下文和报价归档。

## 5. 匹配规则

精确匹配：

- 标准化后完整相等。

编码模糊匹配：

- 用户开启开关后，编码、SKU、型号字段允许包含匹配。

名称匹配：

- 商品名称默认使用包含匹配。

供应商报价表：

- 可以独立产生候选。
- 不要求采购表或聚水潭存在同款商品。

## 8. 开发守则

- 不要随意修改前端既定字段名和卡片布局。
- 新来源应增量加入 `matched_sources`，不要覆盖已有来源。
- 动态成本只用于聚水潭成本波动展示，不要参与采购记录页筛选。
- 空数据不要伪装成采购信息，例如聚水潭供应商不能写入 `supplier_name`。
- Docker 部署必须保留 `brand_item_price/price.db`。
